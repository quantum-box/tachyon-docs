# Worktree管理

Tool Job Workerでは、Git Worktreeを使用してコード編集を安全に行います。

## 概要

Worktreeを使用することで、以下のメリットがあります：

1. **分離された作業環境**: mainブランチに影響を与えずに編集
2. **並行作業**: 複数のJobを同時に処理可能
3. **安全なクリーンアップ**: 失敗したJobの変更を簡単に破棄
4. **PR作成**: 各Jobの変更を独立したPRとして提出

## 運用モード

| モード | 想定環境 | リポジトリの持ち方 | 目的 |
|---|---|---|---|
| Host Mount | ローカル開発 | ホストのリポジトリをマウント | hot reload回避のためWorktreeを活用 |
| Container Clone | 本番ECS | コンテナ内でクローン | 本番環境での完全分離 |

> Docker上でホストの **git worktreeディレクトリ** をマウントすると、`.git` がファイルになり、コンテナ内で参照できないパスを指すため `git worktree add` が失敗することがある。

## Worktreeライフサイクル

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Creating  │────▶│    Ready    │────▶│   Working   │
└─────────────┘     └─────────────┘     └─────────────┘
                                               │
                    ┌──────────────────────────┼──────────────────────────┐
                    │                          │                          │
                    ▼                          ▼                          ▼
             ┌─────────────┐           ┌─────────────┐           ┌─────────────┐
             │  Completed  │           │   Merged    │           │   Failed    │
             └─────────────┘           └─────────────┘           └─────────────┘
                    │                          │                          │
                    │                          │                          │
                    ▼                          ▼                          ▼
             ┌─────────────────────────────────────────────────────────────┐
             │                        Cleanup                              │
             └─────────────────────────────────────────────────────────────┘
```

### ステータス説明

| ステータス | 説明 |
|-----------|------|
| Creating | Worktree作成中 |
| Ready | 編集可能（Tool Job待ち） |
| Working | Tool Job実行中 |
| Completed | 作業完了（マージ待ち/PR作成済み） |
| Merged | mainブランチにマージ済み |
| Failed | 作業失敗 |
| Cleanup | クリーンアップ中 |

## Tool Jobでの使用

### use_worktreeパラメータ

Tool Job作成時に`use_worktree: true`を指定すると、自動的にWorktreeが作成されます。

```bash
POST /v1/agent/tool-jobs

{
  "prompt": "Fix the typo in README.md",
  "provider": "ClaudeCode",
  "metadata": {
    "use_worktree": true,
    "auto_merge": false
  }
}
```

### 処理フロー

1. **Worktree作成**
   ```bash
   git worktree add /repos/project.worktree/worktree-<job_id> -b agent/task-<job_id>
   ```

2. **Coding Agent実行**
   - 作業ディレクトリをWorktreeに設定
   - Claude Code / Codex でコード編集

3. **コミット**
   ```bash
   git add -A
   git commit -m "feat: <summary generated by agent>"
   ```

4. **PR作成（auto_merge=false時）**
   ```bash
   git push -u origin agent/task-<job_id>
   gh pr create --title "..." --body "..."
   ```

5. **直接マージ（auto_merge=true時）**
   ```bash
   git checkout main
   git merge agent/task-<job_id>
   git push origin main
   ```

6. **クリーンアップ**
   ```bash
   git worktree remove /repos/project.worktree/worktree-<job_id>
   git branch -d agent/task-<job_id>
   ```

## コンフリクト処理

### コンフリクト検出

PR作成前にmainブランチとのコンフリクトをチェックします。

```rust
// packages/agents/src/worktree/manager.rs
pub async fn check_for_conflicts(&self, worktree_path: &Path) -> Result<Vec<String>> {
    // git merge --no-commit --no-ff main
    // コンフリクトがあればファイル一覧を返す
}
```

### コンフリクト時の動作

コンフリクトが検出された場合：

1. **callbackに通知**
   ```json
   {
     "tool_job_id": "tj_01xyz",
     "status": "SUCCEEDED",
     "has_conflicts": true,
     "conflicts": ["src/main.rs", "Cargo.toml"],
     "pr_url": "https://github.com/.../pull/123"
   }
   ```

2. **PRは作成される**（手動でコンフリクト解決が必要）

3. **Worktreeは保持される**（コンフリクト解決後にクリーンアップ）

## 自動クリーンアップ

### TTL（Time To Live）

Worktreeにはデフォルトで24時間のTTLが設定されています。

```yaml
# 環境変数
WORKTREE_DEFAULT_TTL: 24  # hours
```

### クリーンアップUsecase

期限切れWorktreeの定期クリーンアップ：

```rust
// packages/agents/src/usecase/cleanup_expired_worktrees.rs
pub struct CleanupExpiredWorktreesInteractor {
    worktree_repository: Arc<dyn WorktreeRepository>,
    worktree_manager: WorktreeManager,
}
```

### 長時間Worktreeのアラート

12時間以上経過したWorktreeに対してアラートを発行：

```rust
// packages/agents/src/usecase/alert_long_running_worktrees.rs
pub struct AlertLongRunningWorktreesInteractor {
    worktree_repository: Arc<dyn WorktreeRepository>,
    threshold_hours: u64,  // デフォルト: 12
}
```

## データベーススキーマ

```sql
CREATE TABLE agent_worktrees (
    id VARCHAR(29) PRIMARY KEY,
    path VARCHAR(512) NOT NULL,
    branch_name VARCHAR(255) NOT NULL,
    tool_job_id VARCHAR(29),
    status VARCHAR(20) NOT NULL DEFAULT 'creating',
    operator_id VARCHAR(29) NOT NULL,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    expires_at TIMESTAMP,

    INDEX idx_worktrees_status (status),
    INDEX idx_worktrees_operator (operator_id),
    INDEX idx_worktrees_tool_job (tool_job_id),
    INDEX idx_worktrees_expires (expires_at)
);
```

## トラブルシューティング

### Worktree作成に失敗する

```
fatal: '<branch>' is already checked out at '<path>'
```

**対処**: 既存のWorktreeを確認・削除
```bash
git worktree list
git worktree remove <path>
```

### Dockerでホストworktreeをマウントした場合

```
fatal: not a git repository: /path/to/worktree
```

**原因**: `.git` がファイルになり、ホストパスを指すためコンテナ内で解決できない。  
**対処**:

1. **メインリポジトリをマウント**する（`.git` がディレクトリのまま）  
2. **コンテナ内でリポジトリをクローン**して作業する  
3. Worktreeが使えない場合は、**graceful fallback** で通常ブランチ作業に切り替える

### Worktreeが削除できない

```
fatal: '<path>' is a main working tree
```

**対処**: メインのworking treeは削除不可。ブランチを削除する場合は先に他のブランチにcheckout

### ディスク容量不足

**対処**:
1. 期限切れWorktreeを手動削除
2. `WORKTREE_DEFAULT_TTL`を短くする
3. ディスク容量を増やす
