# Structured Logging Guidelines

## Overview

This document defines the best practices for structured logging in the tachyon-apps project. Structured logging enables efficient log searching, filtering, and analysis in production environments.

## Core Principles

### 1. Use Structured Fields, Not String Interpolation

**Bad:**
```rust
tracing::info!("Processing request for user {} in tenant {}", user_id, tenant_id);
```

**Good:**
```rust
tracing::info!(
    user_id = %user_id,
    tenant_id = %tenant_id,
    "Processing request"
);
```

### 2. Standard Field Names

Use consistent field names across the codebase:

| Field | Description | Example |
|-------|-------------|---------|
| `request_id` | Unique request identifier | `a1b2c3d4-...` |
| `user_id` | User identifier | `us_01hs2...` |
| `operator_id` | Operator/tenant identifier | `tn_01hj...` |
| `chatroom_id` | Chatroom identifier | `cr_01...` |
| `execution_id` | Agent execution identifier | `exec_01...` |
| `duration_ms` | Operation duration in milliseconds | `123` |
| `error_code` | Error code for categorization | `BAD_REQUEST` |
| `error_type` | Error category (server/client) | `server` |

### 3. Log Levels

| Level | Use Case | Example |
|-------|----------|---------|
| `ERROR` | Server errors (5xx), unexpected failures | Database connection lost, internal server error |
| `WARN` | Client errors (4xx), recoverable issues | Invalid input, permission denied, not found |
| `INFO` | Significant business events | Request started, operation completed, state changes |
| `DEBUG` | Detailed debugging information | Variable values, intermediate states |
| `TRACE` | Very detailed tracing (rarely used) | Function entry/exit, loop iterations |

### 4. Using `#[tracing::instrument]`

For function-level tracing:

```rust
#[tracing::instrument(
    name = "usecase::execute",
    skip(self, input),
    fields(
        chatroom_id = %input.chatroom_id,
        user_id = %input.executor.get_id(),
    )
)]
async fn execute(&self, input: InputData) -> Result<Output> {
    // Function body
}
```

**Key options:**
- `name`: Custom span name (use `module::function` format)
- `skip`: Skip large/sensitive parameters from being recorded
- `skip_all`: Skip all parameters
- `fields`: Add custom structured fields to the span

### 5. Request ID Propagation

All HTTP requests automatically include a `request_id` in the span. This ID:
- Is generated by `SetRequestIdLayer` as a UUID
- Is available in the `x-request-id` header
- Is automatically included in all child spans

To access the request ID in handlers:
```rust
let request_id = request
    .headers()
    .get("x-request-id")
    .and_then(|v| v.to_str().ok())
    .unwrap_or("unknown");
```

### 6. Error Logging

Errors are automatically logged at appropriate levels:
- **5xx errors (InternalServerError, ServiceUnavailable)**: `ERROR` level
- **4xx errors (BadRequest, Unauthorized, Forbidden, NotFound, etc.)**: `WARN` level

When logging errors manually:
```rust
// Server error - use ERROR
tracing::error!(
    error_code = "INTERNAL_SERVER_ERROR",
    message = %e,
    "Database query failed"
);

// Client error - use WARN
tracing::warn!(
    error_code = "BAD_REQUEST",
    message = %e,
    "Invalid input provided"
);
```

## Log Query Examples

### CloudWatch Logs Insights

```sql
-- Find all errors for a specific request
fields @timestamp, @message
| filter request_id = "a1b2c3d4-..."
| sort @timestamp desc

-- Count errors by type
fields error_type, error_code
| filter @message like /error/
| stats count(*) by error_type, error_code

-- Find slow requests (> 1000ms)
fields request_id, path, duration_ms
| filter duration_ms > 1000
| sort duration_ms desc
```

### Datadog Log Search

```
@request_id:a1b2c3d4-* @error_type:server
@operator_id:tn_01hj* @level:ERROR
@execution_id:* @duration_ms:>1000
```

## Implementation Checklist

- [ ] Use structured fields instead of string interpolation
- [ ] Include `request_id` where available
- [ ] Use appropriate log levels (ERROR for 5xx, WARN for 4xx)
- [ ] Add `#[tracing::instrument]` to public methods
- [ ] Include business-relevant fields (user_id, operator_id, etc.)
- [ ] Avoid logging sensitive data (passwords, tokens, PII)
