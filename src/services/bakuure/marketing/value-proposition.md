# バクうれ（bakuure）価値明文化ドキュメント

## このドキュメントの目的

バクうれが「何者で」「誰の」「どんな問題を」「どう解決するか」を明確にする。
社内の意思統一、外部への説明、開発の優先順位判断の基盤とする。

---

## バクうれとは何か

### 定義

**バクうれは「販売オペレーションの統合バス」である。**

各企業が導入している CRM・決済・配送などの SaaS を、販売というドメインの知識に基づいて接続し、
マスタデータの一元管理、販売フローの自動化、SaaS間のリアルタイム同期を行うミドルウェアサービス。

### バクうれが「ない」世界

| SaaS | 担当者の操作 |
|------|------------|
| HubSpot | 営業が案件を作成 → 受注ステータスに変更 |
| _(手作業)_ | 営業がSlackで「受注しました」と連絡 |
| _(手作業)_ | 経理がStripeで請求書を手動作成 |
| Stripe | 経理が決済完了を確認 |
| _(手作業)_ | 経理がSlackで「入金確認しました」と連絡 |
| _(手作業)_ | CSがHubSpotの案件を確認して納品手配 |
| 配送システム | CSが配送依頼を手動入力 |
| _(手作業)_ | CSがHubSpotに「納品完了」を手動更新 |

**結果: 1件の受注に4〜5人が関与し、3〜5回の手作業コミュニケーションが発生。**

### バクうれが「ある」世界

| ステップ | 実行者 | 処理内容 |
|---------|--------|---------|
| 受注 | 営業（HubSpot） | 案件を受注ステータスに変更 |
| ↓ | バクうれ（自動） | HubSpot Webhook → バクうれに通知 |
| ↓ | バクうれ（自動） | 見積書を自動生成、Stripeに決済リクエスト |
| ↓ | バクうれ（自動） | 決済完了 → 納品手配を自動実行 |
| ↓ | バクうれ（自動） | 納品完了 → HubSpotのDealを自動更新 |
| 確認 | 営業（HubSpot） | Dealが「納品完了」になっていることを確認 |

**結果: 営業が1回操作するだけ。残りは全自動。**

## 価値の構造

バクうれの価値は3層で構成される。

```
┌─────────────────────────────────────────┐
│  第3層: ビジネス価値（経営者が得る価値）     │
│  「売上プロセスの全体最適化」               │
│  - リードタイム短縮                        │
│  - 人的ミス削減                            │
│  - 販売データのリアルタイム可視化            │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────┴──────────────────────┐
│  第2層: 業務価値（SOA管理者が得る価値）      │
│  「部署間連携の自動化」                     │
│  - SaaS間のデータ同期が自動               │
│  - 業務フローがルールベースで実行           │
│  - マスタデータが一元管理される             │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────┴──────────────────────┐
│  第1層: 機能価値（現場担当者が得る価値）     │
│  「自分のSaaSだけ使えばいい」              │
│  - 二重入力不要                            │
│  - 他部署への連絡不要                      │
│  - 情報が常に最新                          │
└─────────────────────────────────────────┘
```

## 機能から価値への変換テーブル

### マスタデータ一元管理

| 機能 | ユーザーの行動変化 | ビジネスインパクト |
|------|-----------------|-----------------|
| 商品マスタ管理（Product/Plan/Option） | 商品登録を1箇所で行うだけで全SaaSに反映 | 二重入力コスト削減、データ不整合ゼロ |
| 顧客マスタ管理（Client） | 顧客情報を1箇所で更新するだけ | CRM/決済/配送の顧客情報が常に一致 |
| オブジェクトマッピング | SaaS間のID紐付けを意識しなくていい | 「どのSaaSの情報が正しいか」問題が消滅 |
| 価格管理（RevenueCycle） | 月額/年額の料金体系を一元定義 | 価格変更時の全SaaS反映が即時 |

### 販売フロー自動化

| 機能 | ユーザーの行動変化 | ビジネスインパクト |
|------|-----------------|-----------------|
| セルフサービス注文（SelfServiceOrder） | 顧客が自分でシミュレーション→注文→決済 | 24/365の自動販売、営業コスト削減 |
| 見積→決済連携（ProcessQuotePaymentCheckout） | 見積承認→決済が自動実行 | 受注→入金のリードタイム短縮 |
| デジタル納品（DeliverSoftware） | 決済完了→即座に納品 | 納品リードタイムがゼロに |
| 配送手配（ShippingDestination） | 決済完了→配送先に自動手配 | 物流担当の手作業削減 |
| サブスク管理（Subscription） | 更新/解約が自動処理 | 解約漏れ・更新漏れゼロ |

### SaaS間リアルタイム同期

| 機能 | ユーザーの行動変化 | ビジネスインパクト |
|------|-----------------|-----------------|
| HubSpot Webhook連携 | 営業がCRMで操作→裏側が自動連動 | 営業はCRMだけに集中できる |
| Deal自動化（DealAutomation） | イベント発生→CRM/決済/配送が自動更新 | 部署間の手動連絡がゼロに |
| 商品同期（CRM ↔ バクうれ） | HubSpotの商品情報が自動で最新に | CRM上の商品情報が常に正確 |
| OAuth連携（Square/HubSpot） | ワンクリックでSaaS接続 | IT管理者の導入工数削減 |

## ユーザーストーリー

### Story 1: IT管理者が商品マスタを統合する

> 「うちの会社では商品情報がHubSpot、Stripe、社内Excelの3箇所にあって、
> 価格改定のたびに3箇所を手動で更新していた。
> バクうれを導入してからは、バクうれの管理画面で1回更新すれば、
> HubSpotの商品もStripeのPriceも自動で更新される。
> 二重入力のミスがなくなったし、月末の突合作業も不要になった。」

**対応機能:**
- 管理画面: 商品CRUD（bakuure-admin-ui）
- バックエンド: CreateProduct / UpdateProduct / DeleteProduct
- 同期: CRM CreateProduct + ProviderObjectMapping

### Story 2: 営業マネージャーが販売フローを自動化する

> 「以前は営業が受注したら、Slackで経理に連絡して、
> 経理がStripeで請求書を作って、入金確認したらCSに連絡して…
> と4〜5人が順番にバトンタッチしていた。
> バクうれを入れてからは、営業がHubSpotで受注ステータスにするだけ。
> 請求→決済→納品→CRM更新まで全部自動で回る。
> 受注から納品まで2日かかっていたのが、デジタル納品なら即日。」

**対応機能:**
- Webhook: HubSpot → バクうれ通知
- 自動化: DealAutomation（受注イベント→決済フロー）
- 決済: ProcessQuotePaymentCheckout
- 納品: DeliverSoftware / DeliverInitialOrder

### Story 3: 顧客がセルフサービスで購入する

> 「深夜にサービスを比較検討していて、料金シミュレーターで見積もりを出した。
> そのまま会社情報を入力して、クレジットカードで決済したら、
> すぐにサービスが使えるようになった。
> 営業担当と商談する必要がなかったので、検討開始から30分で導入完了。」

**対応機能:**
- 顧客向けUI: 料金シミュレーター（ProductSimulator）
- 注文フロー: 見積確認→情報入力→決済→完了
- バックエンド: SelfServiceOrder
- 決済: Stripe PaymentCheckoutForm
- 納品: DeliverSoftware

### Story 4: 経営者が販売プロセス全体を把握する

> 「今月の受注はいくらで、そのうち決済完了がいくらで、
> 納品済みがいくらか。以前は週次ミーティングで口頭報告を聞くしかなかった。
> バクうれのダッシュボードでリアルタイムに全体が見える。
> 決済が滞留しているケースにすぐ気づけるようになった。」

**対応機能:**
- 📝 Analytics ダッシュボード（UI未実装 — 将来機能）
- ✅ データ基盤は存在（全トランザクションがバクうれを通過するため集計可能）

## バクうれが「やらないこと」

ポジショニングを明確にするために、バクうれが意図的にやらないことも定義する。

| やらないこと | 理由 | 代わりに |
|------------|------|---------|
| CRMの置き換え | 営業担当はHubSpot/SFに慣れている。置き換えはコスト高 | HubSpotをそのまま使い、裏側で連携 |
| 決済処理の自前実装 | Stripeが圧倒的に優秀。再発明は無意味 | Stripe/Squareに決済を委譲 |
| 在庫管理の自前実装 | 専門SaaSが存在する | 在庫管理SaaSと連携（Stockmind等） |
| EC（消費者向け販売） | B2CはShopifyが強い | B2B販売プロセスに特化 |
| BIツール | Looker/Metabaseが優秀 | 販売プロセスのオペレーション可視化に集中 |

**原則: 各SaaSの「点」の機能は既存SaaSに任せ、バクうれは「線」を引くことに集中する。**

## 価値を実感できる最小シナリオ（MVP体験）

バクうれの価値を最短で実感してもらうためのシナリオ。

### 5分で分かるデモシナリオ

```
Step 1（1分）: 管理画面で商品を登録
  → HubSpotに自動で商品が作成されることを確認

Step 2（1分）: 料金シミュレーターで見積作成
  → 顧客視点で商品を選び、料金を確認

Step 3（1分）: セルフサービスで注文
  → 会社情報入力→Stripe決済→完了

Step 4（1分）: HubSpotを確認
  → Deal/Quote/Companyが自動作成されていることを確認

Step 5（1分）: Stripeを確認
  → Customer/Payment/Subscriptionが自動作成されていることを確認
```

**このデモで伝わること:**
> 「商品を1箇所で登録したら、顧客がセルフサービスで買えて、
>  CRMにも決済サービスにも全部自動で反映される」

## 指標（KPI）

バクうれの価値を測定するための指標。

### 効率化指標

| 指標 | 導入前 | 導入後（目標） |
|------|--------|-------------|
| 受注→納品リードタイム | 2〜5営業日 | デジタル: 即時 / 物理: 1営業日 |
| 1受注あたりの関与者数 | 4〜5人 | 1人（営業のみ） |
| マスタデータ更新にかかる時間 | 30分（3SaaS手動更新） | 3分（1箇所で完了） |
| 月次データ突合作業 | 8時間 | 0時間（自動同期） |

### 売上指標

| 指標 | 意味 |
|------|------|
| セルフサービス注文率 | 人手を介さない注文の割合 |
| 注文完了率（CVR） | シミュレーター→注文完了の転換率 |
| 平均受注リードタイム | 商談開始→決済完了の平均時間 |
| SaaS間データ同期成功率 | 同期エラーが発生しない割合 |

## 今後の価値拡張方向

### 短期（実装済み機能の強化）

1. **同期の可視化**: オブジェクトマッピングの管理画面（今は裏側で動いているが見えない）
2. **自動化ルールの管理画面**: DealAutomationの設定をUIから行えるようにする
3. **Analyticsダッシュボード**: 販売プロセス全体のファネル可視化

### 中期（新規価値の追加）

4. **ルールエンジン**: ノーコードでビジネスルールを定義（「10万円以上は承認が必要」等）
5. **マルチSaaS対応**: Salesforce、freee、ロジレス等への接続拡大
6. **通知・アラート**: 異常検知（決済滞留、同期エラー等）の自動通知

### 長期（プラットフォーム化）

7. **マーケットプレイス**: SaaS連携コネクタのマーケットプレイス
8. **AI最適化**: 販売フローの自動最適化提案
9. **業界テンプレート**: SaaS企業向け、製造業向け等のプリセット
