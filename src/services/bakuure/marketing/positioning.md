# バクうれ（bakuure）ポジショニングドキュメント

## 一言で言うと

> **バクうれ = B2B販売プロセスの統合オーケストレーター**
>
> 各SaaSは現場担当者の道具。バクうれはそれらをビジネスルールに基づいて連携させる「中間層」。
> Revenue Operations（RevOps）の思想を、中小〜中堅企業が使える形で実現する。

---

## 市場環境

### RevOps市場の急成長

Revenue Operations（RevOps）は、営業・マーケ・CSを横断してデータとプロセスを統合する概念として急速に注目されている。

| 指標 | 数値 |
|------|------|
| RevOps市場規模（2032年予測） | **217億ドル** |
| アジア太平洋CAGR | **19.1%**（最も急成長） |
| ツール統合に満足している専門家 | わずか**20%** |
| テックスタックのROIが「平均以下」 | **47%** |

**核心的な課題**: RevOps専門家の80%がツール統合に不満を持ち、CRMとBilling/Financeの間に構造的な断絶がある。

### Quote-to-Cash（Q2C）市場

見積→決済の自動化市場も成長中だが、決定的なギャップがある。

| 指標 | 数値 |
|------|------|
| Q2C市場規模（2024年） | 11.7億ドル |
| Q2C市場規模（2032年予測） | **27.5億ドル**（CAGR 12.4%） |

- 高成長SaaS企業の**93%**が、決済遅延・デフォルト・契約後の摩擦でARRの10%以上を失っている
- CRM/契約/請求システム間のデータ移動が依然として手作業に依存

- 62%以上の企業がクラウドベースの統合フレームワークを採用
- 54%以上がリアルタイムデータ交換を優先

## 市場の構造的ギャップ（バクうれの機会）

### ギャップ1: CRM → Billing → Delivery を一気通貫で統合するプラットフォームが存在しない

現在の市場では「CRM」「Billing/Payment」「Delivery/Fulfillment」がそれぞれ独立したサイロとして存在している。Quote-to-Cash-to-Deliveryの全プロセスを1つのプラットフォームで自動化するソリューションは事実上ない。

### ギャップ2: 中小企業向けのRevOpsプラットフォームが不在

- Salesforce Revenue Cloud → 大企業向け（$30-75/user/月 + 実装コスト莫大）
- HubSpot Operations Hub → CRM中心で決済・配送の深い統合なし
- MuleSoft / Workato → 年間$62,000〜$500,000、IT専任チーム必須

**中小〜中堅企業が手頃な価格でEnd-to-EndのRevOpsを実現する手段がない。**

### ギャップ3: 日本市場向けのB2B販売プロセス自動化が空白

- **LayerX（バクラク）**: バックオフィス特化（請求書処理・経費精算）。フロントオフィスの販売プロセスは範囲外
- **openpage**: カスタマーサクセスの可視化に特化。販売自動化・決済連携は範囲外
- **楽楽B2B / スマレジEC**: 受発注業務のWeb化だが、CRM連携・マスタ同期は弱い

**日本の商習慣（請求書払い、納品書、掛け払い）に対応した販売プロセス統合は誰もやっていない。**

### ギャップ4: iPaaSは「接着剤」であって「プラットフォーム」ではない

Zapier/Make/Workatoはシステム間の接着剤として機能するが、販売ドメインのビジネスロジックを内包していない。見積→注文→請求→配送→収益認識というプロセスそのものを「理解」して自動化するソリューションは不在。

## 解決する課題

### 現在のSaaSエコシステムの構造的欠陥

企業がSaaSを導入すると、以下の3層構造が生まれる。

```
┌─────────────────────────────────────────────────┐
│  Layer 3: 経営層 / マネジメントレイヤー              │
│  BI、ダッシュボード、KPI                            │
│  → データを「見る」ことはできる                       │
│  → しかし操作・介入の手段がない                       │
└────────────────────┬────────────────────────────┘
                     │
                     │  ★ ここが空白地帯 ★
                     │
                     │  「SOA管理者」＝部署間連携の
                     │   リアルタイム同期・業務フロー管理
                     │   → 誰もやっていない
                     │   → 人手でExcelやSlackで繋いでいる
                     │
┌────────────────────┴────────────────────────────┐
│  Layer 1: 現場担当者向け SaaS                       │
│  営業 → HubSpot / Salesforce                      │
│  経理 → Stripe / Square / freee                   │
│  CS   → Intercom / Zendesk                        │
│  物流 → ロジレス / シッピーノ                         │
│  → 各部署の業務は効率化されている                      │
│  → しかし部署をまたぐ業務フローは断絶している           │
└─────────────────────────────────────────────────┘
```

### 具体的な痛み

| 誰が | 何に困っている | 今どうしている |
|------|--------------|--------------|
| 営業マネージャー | 営業がHubSpotで受注したが、経理がStripeで決済処理するまでタイムラグがある | Slackで「決済お願いします」と手動連絡 |
| 経理担当 | Stripeの入金情報とCRMの案件情報が紐づかない | Excelで突合作業 |
| CS担当 | 顧客が何を買ったかCRMだけでは分からない | 別画面でStripeを開いて確認 |
| 経営者 | 営業→受注→決済→納品の全体フローが見えない | 週次ミーティングで口頭報告 |
| IT管理者 | 各SaaSの商品マスタが同期されていない | 手動で二重入力、CSVインポート |

## バクうれのポジション

### 市場マッピング

```
                    販売ドメイン特化度
                         高
                          │
                          │    ★ バクうれ ★
                          │    B2B販売プロセスの
                          │    統合オーケストレーター
                          │
              HubSpot ────┤──── Shopify
              Salesforce  │     EC特化（B2C）
              CRM特化      │
                          │
         Clay/Apollo ─────┤
         リード生成特化     │
                          │
         Zapier/Make ─────┤──── MuleSoft/Workato
         汎用トリガー連携   │     汎用iPaaS
                          │
                         低
            軽量 ──────────┼────────── 重量
              導入コスト / 統合の深さ
```

### 競合との詳細比較

#### vs Zapier / Make（汎用自動化ツール）

| 観点 | Zapier ($30-104/月) | Make ($19-34/月) | バクうれ |
|------|-----|------|--------|
| 連携の深さ | トリガー→アクションの1:1 | ビジュアルシナリオ | 販売プロセス全体のオーケストレーション |
| ドメイン知識 | なし（汎用、8000+アプリ） | なし（汎用、3000+アプリ） | 販売ドメイン特化（見積、決済、納品を理解） |
| マスタ管理 | なし | なし | 商品・顧客マスタの一元管理 |
| UI | 自動化フロー設定のみ | 自動化フロー設定のみ | 顧客向け販売UI + 管理画面 |
| 弱点 | 複雑な分岐・大量データに不向き | 学習曲線が急 | SaaS連携数はまだ少ない |

#### vs HubSpot Operations Hub（CRM拡張）

| 観点 | HubSpot Ops ($20-800/月) | バクうれ |
|------|------------|--------|
| 決済実行 | 米国限定（HubSpot Payments） | Stripe/Square統合（グローバル） |
| 納品管理 | なし | デジタル+物理配送 |
| マスタ管理 | HubSpot内のみ | 複数SaaSへの配信 |
| セルフサービス | なし | 顧客向け注文UI + 料金シミュレーター |
| 弱点 | HubSpotエコシステム内に閉じる | HubSpot依存度が現状高い |

#### vs Salesforce Revenue Cloud（エンタープライズ）

| 観点 | SF Revenue Cloud ($30-75/user/月 + 実装費) | バクうれ |
|------|------------|--------|
| ターゲット | 大企業 | 中小〜中堅企業 |
| 導入コスト | 数百万〜数千万円 | 低コスト |
| 柔軟性 | SFエコシステムに完全依存 | 既存SaaSを活かしたまま統合 |
| 日本対応 | 日本の商習慣への対応が弱い | ネイティブ対応可能 |

#### vs 日本国内プレイヤー

| 観点 | LayerX（バクラク） | openpage | バクうれ |
|------|---|---|--------|
| 領域 | バックオフィス（請求書・経費） | カスタマーサクセス | **フロントオフィス（販売プロセス）** |
| CRM連携 | なし | 閲覧用ログのみ | HubSpot双方向同期 |
| 決済連携 | なし | なし | Stripe/Square統合 |
| 配送連携 | なし | なし | デジタル+物理配送 |

### バクうれの独自ポジション

**「B2B販売プロセス特化の統合オーケストレーター」**

- Zapierより深い — ビジネスルールを理解し、販売フロー全体を管理する
- ERPより軽い — 既存SaaSの運用を壊さず、上に乗る
- CRMより広い — 決済・納品・在庫まで一気通貫
- **日本の商習慣にネイティブ対応** — 請求書払い、納品書、掛け払いを前提とした設計

## 3つのコアバリュー

### Value 1: マスタデータの一元管理（Single Source of Truth）

```
バクうれ（マスタ）
├─→ HubSpot: 商品として同期
├─→ Stripe: Price/Productとして同期
├─→ 自社EC: カタログとして同期
└─→ 在庫管理: SKUとして同期
```

**何が嬉しいか:**
- 商品を1箇所で登録すれば全SaaSに反映される
- 価格変更も1箇所で済む
- 「どのSaaSが正なのか分からない」問題が消える

**実装状況:**
- ✅ 商品マスタ（Product / Plan / Option の3タイプ）
- ✅ 顧客マスタ（Client）
- ✅ オブジェクトマッピング（bakuure ID ↔ 各SaaS ID）
- ✅ HubSpot双方向同期
- ✅ Stripe Product/Price 連携

### Value 2: 販売プロセスのオーケストレーション

```
見積作成 → 承認 → 決済 → 納品 → CRM更新
   │         │       │       │        │
   │         │       │       │        └─ HubSpot Deal更新
   │         │       │       └─ デジタル納品 or 配送手配
   │         │       └─ Stripe決済実行
   │         └─ ビジネスルール判定
   └─ 商品選択 + 料金シミュレーション
```

**何が嬉しいか:**
- 「営業→経理→CS→物流」の部署リレーが自動化される
- 各ステップの実行は既存SaaSが行い、バクうれは「いつ・誰が・何をするか」を管理
- 人が介在しないセルフサービス販売も同じフローで実現

**実装状況:**
- ✅ セルフサービス注文（SelfServiceOrder）
- ✅ 見積→決済→納品の一気通貫フロー
- ✅ イベント駆動の自動化（DealAutomation）
- ✅ 料金シミュレーター（顧客向けUI）
- 🔄 承認フロー（未実装）
- 🔄 条件分岐ルール（部分的）

### Value 3: SaaS間のリアルタイム同期

```
営業がHubSpotで受注
    │
    ├─→ バクうれにWebhookで通知
    │       │
    │       ├─→ Stripeに決済リクエスト
    │       ├─→ 在庫引当
    │       └─→ 配送手配
    │
    └─ 営業はHubSpotだけ見ていればいい
```

**何が嬉しいか:**
- 各担当者は自分のSaaSだけ使い続ければいい
- 裏側でバクうれが全SaaSのデータを同期する
- 「あの情報はどのSaaSを見ればいいの？」が消える

**実装状況:**
- ✅ HubSpot Webhook受信（双方向）
- ✅ イベント駆動のオブジェクト同期
- ✅ ProviderObjectMapping（ID紐付け管理）
- 🔄 Square双方向同期（OAuth接続のみ）
- 📝 汎用Webhook（Stripe Webhook等）

## ターゲットユーザー

### Primary: SOA管理者（部署間連携の責任者）

- 役職: 営業企画、事業推進、DX推進、IT管理者
- 規模: 従業員50〜500名のB2B企業
- 状態: 3〜5個のSaaSを導入済みだが連携に苦戦している
- 痛み: 「SaaSを入れたのに、部署間の連携は相変わらず手作業」

### Secondary: 経営者 / マネジメント層

- 痛み: 営業→決済→納品の全体像が見えない
- バクうれで得るもの: 販売プロセス全体のリアルタイム可視化

### Tertiary: 現場担当者

- 痛み: 複数のSaaSを行き来する手間
- バクうれで得るもの: 自分のSaaSだけ使えば裏側が自動同期される安心感

## メッセージング

### エレベーターピッチ（30秒）

> 「SaaSを複数導入したのに、部署間の連携は相変わらず手作業になっていませんか？
> バクうれは、HubSpotやStripeなどの既存SaaSをそのまま使いながら、
> 商品マスタの同期、見積→決済→納品の自動化、リアルタイムのデータ連携を実現する
> B2B販売プロセスの統合オーケストレーターです。
> 各担当者は自分のSaaSだけ使い続ければOK。裏側の連携はバクうれが全部やります。」

### タグライン案

1. **「SaaSの間を、つなぐ。」**
2. **「売れる仕組みを、自動で回す。」**
3. **「営業も経理も、自分のツールだけでいい。」**
4. **「販売プロセスの全自動化、始まる。」**

### キーメッセージ（3本柱）

1. **つなぐ** — 商品・顧客データを1箇所で管理し、全SaaSに自動同期
2. **回す** — 見積→決済→納品の販売フローをビジネスルールで自動実行
3. **任せる** — 各担当者は自分のSaaSだけ。部署間連携はバクうれにお任せ
