# 価格の不一致分析

## 問題の発見

既存のクレジット単位システムドキュメントと、実際のプロバイダー価格に大きな不一致があることが判明しました。

## 具体的な不一致例

### Claude Opus 4
| ソース | Input価格 | 実際のUSD | クレジット換算 |
|--------|----------|-----------|---------------|
| 既存ドキュメント | 0.02クレジット/token | $0.0002/token | - |
| 実際のAPI価格 | - | $0.000015/token | 0.0015クレジット/token |
| **差分** | - | **13.3倍** | **13.3倍** |

### Claude Sonnet 4
| ソース | Input価格 | 実際のUSD | クレジット換算 |
|--------|----------|-----------|---------------|
| 既存ドキュメント | 0.005クレジット/token | $0.00005/token | - |
| 実際のAPI価格 | - | $0.000003/token | 0.0003クレジット/token |
| **差分** | - | **16.7倍** | **16.7倍** |

### GPT-4.1
| ソース | Input価格 | 実際のUSD | クレジット換算 |
|--------|----------|-----------|---------------|
| 既存ドキュメント | 0.015クレジット/token | $0.00015/token | - |
| 実際のAPI価格 | - | $0.000002/token | 0.0002クレジット/token |
| **差分** | - | **75倍** | **7.5倍** |

## 結論

これらの価格差は**誤りです**。ハードコードされている価格は実際のAPI価格と一致していなければなりません。

### 原因
- 古い価格データがドキュメントに残っている
- AIモデルの価格下落に追従していない
- 単位の混同やドキュメントの更新漏れ

### 正しいアプローチ
- プロバイダーのpricing.rsファイルの価格が正しい
- これらの実際のAPI価格を使用する
- マークアップはビジネスロジックで別途適用する（ハードコードしない）

## 影響と対応

### 移行への影響
1. **価格の大幅な下落**
   - 既存ユーザーは大幅に安い価格になる
   - 収益への影響

2. **システムの整合性向上**
   - 実際のAPI価格と一致することで透明性が向上
   - 価格変更時の追従が容易

### 必要な対応
1. **ハードコードされた価格を実際のAPI価格に修正**
   - `/docs/src/tachyon-apps/billing/credit-unit-system.md`を更新
   - 実際のpricing.rsファイルの価格を使用

2. **マークアップの適用方法を変更**
   ```rust
   // ❌ 古い方法（ハードコード）
   let claude_price = 0.02; // クレジット/token
   
   // ✅ 新しい方法（実際のAPI価格 + 動的マークアップ）
   let api_price = pricing_provider.get_price("claude-4-sonnet");
   let markup_rate = config.get_markup_rate();
   let final_price = api_price * markup_rate;
   ```

3. **段階的移行**
   - 実際のAPI価格に戻す
   - 必要に応じて設定でマークアップを適用
