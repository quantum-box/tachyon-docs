## 概要

ExecuteAgentが保存するチャットメッセージ履歴をGetAgentHistoryで正しく復元できるよう、代表的な記録形式を再現した単体テストを追加しテストカバレッジを高める。

## 背景・目的

- エージェント実行時にCommandStackが保存するメッセージは`<task>`タグや`[tool] Result:`などのプレーンテキスト表現を用いており、GetAgentHistoryではそれらを`AgentChunk`へ変換している。
- 従来の単体テストは単純なユーザー/アシスタントメッセージのみを想定しており、ExecuteAgent固有のメッセージパターンを網羅していない。
- 実際の履歴を模したテストを追加することで、履歴復元ロジックの退行を早期に検知できるようにする。

## 詳細仕様

### 機能要件

1. ExecuteAgentが初期化時に保存する`<task>`付きユーザーメッセージを再現し、GetAgentHistoryでタグ除去後のテキストへ変換されることを検証する。
2. `[tool] Result:`で始まるユーザーメッセージ、および`<attempt_completion>`タグを含むアシスタントメッセージから適切な`AgentChunk`へ復元されることを確認する。
3. 既存の認可チェックと空履歴処理の退行が起きないよう既存ケースを維持・補強する。
4. `<use_mcp_tool>` で表現される MCP ツール呼び出しが ToolCall/ToolCallArgs/Say のシーケンスへ変換されることを検証する。

### 非機能要件

- テストはMockリポジトリを用い、副作用を持たない。
- ドメインロジックを変更せず、テスト実行時間への影響を最小限に抑える。
- 将来的にメッセージフォーマットが追加された際にテストを拡張しやすい構成とする。

## 実装方針

### アーキテクチャ設計

- Clean Architectureのユースケース層に位置するGetAgentHistoryのテストを補強する。
- MockChatMessageRepositoryを通じてExecuteAgent相当の保存結果を再現し、ユースケース内の`messages_to_chunks`変換を検証する。

### 技術選定

- 既存のRustユニットテスト（tokio + async_trait + mockall）を継続利用。
- Message構造体を介してChatMessage生成を行い、実装との乖離を防ぐ。

### TDD（テスト駆動開発）戦略

#### 既存動作の保証
- 既存の空履歴テストを維持し、追加ケースで回帰検知を強化する。

#### テストファーストアプローチ
- ExecuteAgentが保存するフォーマットを洗い出してテストに落とし込み、必要なら補助関数を先に定義してからテストを書く。

#### 継続的検証
- `cargo nextest run -p llms --lib get_agent_history`で対象テストを優先実行し、余裕があれば`mise run test`にも反映する。

### 実施手順
1. **ローカル開発サーバーの確認**

   - 起動していない場合のみ `yarn dev --filter=<app-name>` を実行
   
2. **動作確認レポートの作成**
   - `./verification-report.md` を作成（タスクドキュメントと同じディレクトリ内）
   - 以下のテンプレートを使用：
   ```markdown
   # [機能名] 動作確認レポート
   
   実施日: YYYY-MM-DD
   実施者: @<username>

   - ブラウザ: Chrome/Firefox/Safari
   - 画面サイズ: 1920x1080
   - テストユーザー: test-user@example.com
   
   ## 動作確認結果
   
   ### ✅ 基本動作
   - [x] ページの読み込み完了
     ![ページ読み込み](./screenshots/page-load.png)
   
   ### ❌ エラーケース
   - [ ] 認証エラー時の表示
     - 問題: エラーメッセージが表示されない
     - ![エラー画面](./screenshots/auth-error.png)

   1. 認証エラー時にメッセージが表示されない
   2. モバイル表示でレイアウトが崩れる

   - エラーハンドリングの追加が必要
   ```

3. **Playwright MCPでの確認実施**
   - ブラウザを開く（`mcp__playwright__browser_navigate`）
   - 必要に応じてテストユーザーでログイン
   - 対象ページに遷移
   - チェックリストに従って動作確認を実施

4. **スクリーンショットの取得と保存**
   - 重要な画面や問題のある箇所をスクリーンショット（`mcp__playwright__browser_take_screenshot`）
   - `./screenshots/` ディレクトリに保存（タスクドキュメントと同じ階層）

5. **レポートの完成と共有**
   - 全ての確認項目の結果をレポートに記載
