# セルフサインアップE2E動作確認

## 概要

Tachyon のセルフサインアップから初回ワークスペース利用開始までのユーザーフローを Playwright MCP で検証し、現時点（2025-10-15）での阻害要因やワークアラウンドを整理する。

## 背景・目的

- サインアップ／ワークスペース設定フローに既知の外部依存（HubSpot・Stripeなど）が多く、ローカル環境での再現手順が複雑化している。
- 仕様変更や暫定対応が散発的に入っており、最新手順がドキュメント化されていないためオンボーディング時の不明点が多い。
- 新規ユーザーが即時にプロダクトを利用開始できるかを定期的に確認したい。

## 詳細仕様

### 機能要件

1. Playwright MCP を用いて、新規サインアップ開始から初回ダッシュボードアクセスまでの主要ステップを自動操作で確認する。
2. Email認証やワークスペース設定に必要な外部サービス（Mailinator、GraphQL API など）の前提条件を洗い出し、成功／失敗条件を記録する。
3. フロー中に発生したエラーやブロッカーをログ・スクリーンショット付きで残し、ワークアラウンドまたは今後の対応方針を明文化する。

### 非機能要件

- Playwright MCP 実行時は `tachyon-dev` プラットフォームの既定ベースURLを利用する。
- 認証情報やシードデータはリポジトリで共有されているダミー値を用い、秘密情報を記載しない。
- 検証記録は docs 配下にまとめ、再実行しやすい手順を維持する。

## 実装方針

- タスクドキュメントを進捗管理の中心として更新しながら検証を進める。
- Playwright MCP での操作手順は verification-report.md に詳細を残し、必要なスクリーンショットは `screenshots/` ディレクトリに保管する。
- 既知の制約（HubSpot認証エラー等）が発生した場合は再現条件を明示し、動作確認の達成基準を調整する。

## タスク分解

### フェーズ1: 事前準備 📝
- [ ] 必要な環境変数・サーバー起動状態の確認
- [ ] Mailinator 受信箱の準備とアクセス手順の整理
- [ ] Playwright MCP シナリオの確認・初期化

### フェーズ2: サインアップフロー操作 📝
- [ ] サインアップフォーム入力と送信
- [ ] 認証コード確認と入力
- [ ] ワークスペース設定フォームの完了

### フェーズ3: 利用開始確認 📝
- [ ] 初回ダッシュボード到達の確認（成功 or ブロック状況の記録）
- [ ] 必要なスクリーンショット取得
- [ ] 成否・課題の整理と報告

## テスト計画

- Playwright MCP によるUI操作のログ・スクリーンショットを取得する。
- 失敗したステップについては再試行し、再現性の有無を確認する。
- 代替手順が必要な場合は記録し、次回検証時に活用できるようまとめる。

## 前提条件

- `mise run dev` および `mise run dev-backend` が実行中で、Next.js と GraphQL API がローカルで到達可能であること。
- Playwright MCP から `http://localhost:16000/v1beta/tn_01hjryxysgey07h5jz5wagqj0m` へアクセスできること。
- Mailinator の共有受信箱（例: `tachyon-e2e-signup`）にアクセス可能で、既存メールを削除済みであること。
- GraphQL API 呼び出し時に `Authorization: Bearer dummy-token`, `x-operator-id: tn_01hjryxysgey07h5jz5wagqj0m`, `x-user-id: <シードユーザーまたはサインアップユーザー>` を指定すること。

## リスクと対策

| リスク | 対策 |
| --- | --- |
| HubSpot 連携エラーでプロビジョニングが停止する | 既知事象として報告に記載し、エラー画面・ログを収集する |
| GraphQL エンドポイント未起動でセッション更新に失敗 | `mise run dev-backend` 起動状態を前提条件として明記する |
| 認証メールが届かない | Mailinator の受信箱名を固定し、複数回受信できるようクリア手順を準備する |
