## 概要

セルフサインアップ直後のユーザーでも、ヘッダー右上のテナント切替ドロップダウンに「Library Platform」など自分に割り当てられていないプラットフォームが表示されてしまう。プラットフォームに直接所属していないユーザーからは該当プラットフォームを除外し、オペレーターだけを選択肢として提示する。

## 背景・目的

- 新規登録フロー（2025-10-13 時点）では、`SignInWithPlatform` のレスポンスに含まれる `platforms` が NextAuth セッションへ格納されている。
- GraphQL `User.platforms` リゾルバー（`find_accessible_platforms`）が、ユーザーが所属するオペレーターの親プラットフォームを無条件に合成しており、実際には所属していないプラットフォームまで表示される。
- 結果としてセルフサービス利用者が「Library Platform」を選択できてしまい、遷移先で権限不足や未実装画面に直面する。
- プラットフォームに明示的に割り当てられている管理者（例: Tachyon 開発メンバー）には影響を与えず、誤った候補だけを除外することが目的。

## 詳細仕様

### 機能要件

1. `find_accessible_platforms` は、ユーザーが `_tenant_to_user` で直接紐づくプラットフォーム、または `AdminPolicy` を保持するホスト管理者のみを返す。
2. `SignInWithPlatform` → NextAuth セッション → フロントエンドのドロップダウンに渡る `platforms` 配列から未所属プラットフォームが消える。
3. 受け入れ条件
   - セルフサインアップ済みユーザー（`PLATFORM_OPERATOR_ID` にのみ紐づく）でログインした際、ドロップダウン上で自分のオペレーターのみ表示される。
   - Tachyon 開発用ユーザー（`AdminPolicy` 付与済み）は従来どおりプラットフォーム一覧を切り替えられる。
   - 既存の認可チェック（`authWithCheck`）を通過できるテナントは変化しない。

### 非機能要件

- パフォーマンス: プラットフォーム数は多くないため `get_by_id` の追加呼び出しで問題なし。
- セキュリティ: 不必要なテナント ID を UI から排除し、権限外遷移の試行回数を抑制する。
- 保守性: フィルタリング条件は `PlatformInteractorImpl` に集中させ、将来の判定変更を容易にする。


## 実装方針

### アーキテクチャ設計

- Clean Architecture の usecase 層（PlatformInteractor）でフィルタリングを実施し、プレゼンテーション層への影響を最小化する。
- GraphQL Resolver 側のコードは据え置き、戻り値だけを調整する。
- `SignInWithPlatform` の事前フィルタリングは変更せず互換性を保つ。

### 技術選定

- Rust / tokio / sqlx（既存スタック）
- mockall（ユニットテストで依存を差し替える）
- 新規ライブラリ追加なし。

### TDD（テスト駆動開発）戦略
<!-- リファクタリングタスクの場合は必須セクション -->

#### 既存動作の保証
- `PlatformInteractorImpl` のユニットテストを新設し、未所属プラットフォームが除外されることを確認する。
- ホスト管理者が従来どおりプラットフォームを取得できるケースも合わせて検証する。
- 既存 e2e には影響しないため追加ベンチマークは不要。

#### テストファーストアプローチ
- 先にユニットテストを追加し、Red → Green → Refactor の順で進める。
- カバレッジ指標は CI に委ね、ローカルでは対象モジュールのみ実行。

#### 継続的検証
- `cargo test -p auth` をローカルで実行。
- PR では CI の `mise run ci` が全体を検証。

- `PlatformInteractorImpl::find_accessible_platforms` に、ユーザーが `_tenant_to_user` で直接紐づくプラットフォームのみを返すフィルタを追加。
